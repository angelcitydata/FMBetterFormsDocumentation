# Card Component

The Card component provides a flexible content container for displaying structured information in your FM BetterForms application. It features a clean, accessible layout with title, subtitle, body text, and optional fields, making it perfect for organizing and presenting content in a visually appealing format. {{ className: 'lead' }}

    This component supports structured content display with customizable title and subtitle,
    optional body text, and a flexible field system for organizing related information
    in a consistent, accessible card layout.

---

## Component Properties

### Content Properties

<Properties>
  <Property name="title" type="string">
    The main title displayed at the top of the card. This text appears in bold
    and uses the primary foreground color. Example: `"User Profile"`, `"Order Summary"`.
    Default is empty string.
  </Property>
  <Property name="subtitle" type="string">
    The subtitle displayed below the title. This text appears in smaller font
    size with muted foreground color. Example: `"Personal Information"`, `"Items in cart"`.
    Default is empty string.
  </Property>
  <Property name="bodyText" type="string">
    The main body content of the card. This text appears in regular font size
    with standard foreground color. Only displayed if not empty or whitespace.
    Example: `"Complete your profile to get started"`, `"Review your order details"`.
    Default is empty string.
  </Property>
  <Property name="fields" type="array">
    Array of field objects to display within the card. Each field can contain
    various properties for displaying structured data. Currently supports empty array.
    Example: `[]` (empty array for now, future expansion planned).
    Default is empty array.
  </Property>
</Properties>

## Card Layout Structure

The Card component features a clean, structured layout that organizes content hierarchically. The component automatically adapts its display based on the content provided, showing only relevant sections.

### Content Hierarchy

The card displays content in the following order:

1. **Title** - Main heading displayed at the top in bold text
2. **Subtitle** - Secondary heading below the title in smaller, muted text
3. **Body Text** - Main content area (only shown if content exists)
4. **Fields** - Structured data fields (currently supports empty array)

### Visual Structure

- **Container**: Full-width card with background, border, padding, and rounded corners
- **Header Section**: Contains title and subtitle in a vertical flex layout
- **Body Section**: Contains body text with conditional rendering
- **Fields Section**: Reserved for future field rendering functionality

### Responsive Design

The card component is designed to be responsive and accessible:

- **Full Width**: Cards expand to fill their container width
- **Flexible Height**: Cards adjust height based on content
- **Consistent Spacing**: Standard padding and gap spacing throughout
- **Accessible Typography**: Proper heading hierarchy and text contrast

## Usage Examples

<CodeGroup>
```html title="HTML Component"
<bfcomp name="Card" 
        title="Welcome"
        subtitle="Get started with your account"
        bodyText="Complete your profile to begin using all features.">
```

```javascript title="Model Definition"
  fields: []
```

## Component Schema

The complete component schema for the Card component. Copy and paste this code into the Component Schema section of your BetterForms app.

### Complete Schema

<CodeGroup>
```json title="Component Schema"
{
    "bodyText": "",
    "html": "<div class=\"flex flex-col bg-card text-card-foreground rounded-lg p-4 w-full border border-border\">\n    <div class=\"flex flex-col gap-1\">\n        <h3 class=\"text-foreground font-bold\">{{schema.title}}</h3>\n        <h4 class=\"text-sm text-muted-foreground font-medium\">{{schema.subtitle}}</h4>\n    </div>\n\n    <div v-if=\"!_.isEmpty(_.trim(schema.bodyText))\" class=\"flex my-2\">\n        <span class=\"text-base text-foreground\">{{schema.bodyText}}</span>\n    </div>\n</div>",
    "subtitle": "",
    "title": ""
}
```

```html title="HTML Template"
<div class="flex flex-col bg-card text-card-foreground rounded-lg p-4 w-full border border-border">
    <div class="flex flex-col gap-1">
        <h3 class="text-foreground font-bold">{{schema.title}}</h3>
        <h4 class="text-sm text-muted-foreground font-medium">{{schema.subtitle}}</h4>
    </div>

    <div v-if="!_.isEmpty(_.trim(schema.bodyText))" class="flex my-2">
        <span class="text-base text-foreground">{{schema.bodyText}}</span>
    </div>
</div>
```
</CodeGroup>

### How to Add This Component to Your BetterForms App

1. **Navigate to Components**: In your BetterForms site environment, click the **Components** tab in the left sidebar.

2. **Create New Component**: If the Card component is not already in your components list, click the **+New** button at the top left.

3. **Open Component Schema**: In the new component, click the **`</> Component Schema`** button to access the schema editor.

4. **Replace Schema Code**: Copy the complete schema from the **Component Schema** tab above and replace all the existing code in the Component Schema section.

5. **Open HTML Tab**: Click the **HTML** tab (located directly next to the Component Schema tab) to access the HTML editor.

6. **Replace HTML Code**: Copy the HTML template from the **HTML Template** tab above and replace all the existing code in the HTML section.

   **Note**: This step might feel redundant since the HTML is included as a key/value in the component schema. However, this is sometimes a quirk in BetterForms, where the HTML doesn't render properly until you manually insert the HTML in the HTML tab on the new component.

7. **Save Component**: Click the **Save** button in the top right to save the component for use in your application.

8. **Refresh Pages**: **Important**: The component will not be available for use in any live pages you are currently viewing until you manually refresh the page using your browser's Refresh button. However, the component will be immediately available in the **Page Builder** section of every page after you save and navigate away from the Components section.

## Best Practices

When using the Card component, follow these guidelines for optimal user experience and consistent design.

### Content Guidelines

**✅ Do:**

- Keep titles concise and descriptive (1-5 words)
- Use subtitles to provide context or categorization
- Write clear, scannable body text that adds value
- Maintain consistent terminology across your application
- Use cards to group related information logically

**❌ Don't:**

- Use overly long titles that break the card layout
- Include redundant information between title and subtitle
- Write body text that doesn't provide additional value
- Mix different terminology for similar content types
- Create cards with unrelated or scattered information

### Content Structure

**Title Guidelines:**
- Use descriptive, action-oriented titles when appropriate
- Keep titles consistent with your application's voice
- Avoid generic titles like "Information" or "Details"
- Use title case for consistency

**Subtitle Guidelines:**
- Provide context or categorization information
- Use subtitles to explain the purpose or scope
- Keep subtitles shorter than titles
- Use sentence case for readability

**Body Text Guidelines:**
- Write clear, concise explanations or descriptions
- Use body text to provide additional context
- Keep paragraphs short for better readability
- Only include body text when it adds value

### Layout and Design

**✅ Do:**

- Use cards to group related information
- Maintain consistent spacing between cards
- Align cards with other interface elements
- Consider the visual hierarchy of multiple cards
- Use cards to break up long content sections

**❌ Don't:**

- Create cards that are too wide or too narrow
- Overcrowd pages with too many cards
- Use inconsistent spacing between cards
- Place cards in locations that disrupt content flow
- Create cards with excessive white space

### Accessibility Considerations

**✅ Do:**

- Ensure sufficient color contrast for all text elements
- Test with screen readers to ensure proper reading order
- Use descriptive titles that clearly indicate content
- Maintain consistent heading hierarchy
- Provide meaningful content structure

**❌ Don't:**

- Rely solely on visual styling to convey information
- Create cards with poor color contrast
- Use generic or unclear titles
- Forget to test with assistive technologies
- Create cards that are difficult to scan

### Performance Tips

**✅ Do:**

- Use cards for static or semi-static content
- Consider the total number of cards on a page
- Use appropriate model keys for card data
- Implement efficient data loading for card content

**❌ Don't:**

- Use cards for frequently changing content without proper state management
- Overload pages with too many cards that impact rendering performance
- Use generic model keys that might conflict with other components
- Create cards that require excessive data fetching
